# 零式协约 · 赛车游戏项目拆解 V0.1

> **目标：** 小学中高年级 Scratch 图形化编程课程  
> **课时标准：** 每节 45 分钟  
> **版本：** MVP（最小可玩版本）——完成一次完整比赛即可

---

## 一、第一阶段 MVP 总览

| # | 模块 | 核心能力 |
|---|------|----------|
| 1 | 赛车能移动 | 键盘控制 + 角色运动 |
| 2 | 有赛道 | 背景绘制 + 边界设定 |
| 3 | 起点和终点 | 标记物设计 + 坐标概念 |
| 4 | 完整比赛 | 计时器 + 到达检测 + 反馈 |

四个模块串联后，学生能跑出一辆赛车从起点出发、沿赛道行驶、冲过终点并显示用时——这就是"可玩"。

---

## 二、模块详细拆解

### 模块 1：赛车能移动

| 项目 | 内容 |
|------|------|
| **功能描述** | 学生操控一辆赛车角色，用键盘方向键实现上、下、左、右四个方向的移动。赛车在舞台上自由行驶，不越出舞台范围。 |
| **所需积木块** | **事件：** `当绿旗被点击`<br>**控制：** `重复执行` + `如果…那么`<br>** sensing：** `按键被点击？`（上/下/左/右四个方向）<br>**运动：** `移动 10 步`、`面向方向`、`将 x/y 坐标增加 N`、`移到 x:y:` |
| **预估课时** | **1 课时（45 分钟）** |
| **教学要点** | 1. 坐标系概念：舞台上的 X 轴和 Y 轴，正负方向对应上下左右<br>2. 事件驱动思维：按键 → 检测 → 执行动作 的因果链<br>3. `重复执行` 循环的意义：让程序"一直听键盘"<br>4. 条件判断基础：如果按了上键就往上走 |
| **常见问题** | - 赛车移动太快/太慢 → 调整每次移动的步数（建议起步设 5~10）<br>- 四个方向积木顺序搞混 → 强调"先检测、再执行"<br>- 角色初始坐标不在舞台中央 → 先用 `移到 x:0 y:0` 定位<br>- 不理解为什么需要 `重复执行` 包裹按键检测 → 用"如果不重复，按一下就走一步就停了"来解释 |

---

### 模块 2：有赛道

| 项目 | 内容 |
| **功能描述** | 在舞台上呈现一条可视化的赛道，包含道路主体和两侧边界线。赛车只能在赛道范围内行驶（或至少视觉上让玩家感知到赛道存在）。 |
| **所需积木块** | **外观：** 绘制背景（使用 Scratch 画笔工具或导入赛道图片）<br>**运动：** `如果碰到颜色，就反弹` 或 `移到指定坐标`（用于边界限制）<br>** sensing：** `碰到颜色？`（检测赛车是否冲出赛道）<br>**事件：** `当绿旗被点击`（重置场景） |
| **预估课时** | **1 课时（45 分钟）** |
| **教学要点** | 1. 场景设计思维：先搭舞台再放角色<br>2. 画笔工具基础使用：画直线、设置颜色/粗细（或导入素材的方法）<br>3. 颜色碰撞检测原理：赛车碰到"边界色"会被判定出界<br>4. 游戏世界规则的建立：赛道就是"游戏规则"的可视化 |
| **常见问题** | - 赛道太窄，赛车总出界 → 建议先画宽赛道，后续迭代再缩小<br>- 用画笔画的线和导入的图片颜色不一致 → 教学生用取色器确认颜色代码<br>- "碰到颜色"检测不生效 → 检查是否选中了正确的颜色样本<br>- 孩子想把赛道做得特别漂亮导致超时 → 提醒 MVP 原则：先能跑，再好看 |

---

### 模块 3：起点和终点

| 项目 | 内容 |
|------|------|
| **功能描述** | 在赛道两端分别设置起点标记（如起跑线或绿色旗帜）和终点标记（如格子旗或奖杯）。赛车从起点出发，到达终点时有视觉提示。 |
| **所需积木块** | **外观：** 新增起点/终点角色（可画旗帜、门框等），`切换造型`<br>**运动：** `移到 x:y:`（固定起点和终点坐标）<br>** sensing：** `碰到 [终点]？`<br>**控制：** `如果…那么…否则`<br>**事件：** `当绿旗被点击`（确保起点角色位置正确） |
| **预估课时** | **1 课时（45 分钟）** |
| **教学要点** | 1. 多角色协作：赛车、起点、终点是独立的角色，各自有代码<br>2. 坐标精确定位：起点和终点不能随便放，要放在赛道两端<br>3. 碰撞检测进阶：这次不是"碰到颜色"，而是"碰到另一个角色"<br>4. 游戏流程概念：开始 → 行驶 → 结束 三阶段 |
| **常见问题** | - 起点/终点角色挡住赛车路 → 调整 z-order（移到最前/最后）<br>- 终点检测太灵敏或完全不触发 → 检查终点角色大小和位置<br>- 孩子不理解为什么起点也需要写代码 → 解释"绿旗重置"概念<br>- 混淆 `碰到颜色` 和 `碰到角色` → 用对比实验让两种检测同时出现 |

---

### 模块 4：完整比赛（计时 + 到达终点反馈）

| 项目 | 内容 |
|------|------|
| **功能描述** | 赛车从起点出发时开始计时，到达终点后停止计时并显示总用时。通过弹出对话框或文字提示完成比赛的反馈。可选：允许重新比赛。 |
| **所需积木块** | **变量：** 创建 `计时器` 变量<br>**控制：** `等待直到 [条件]`<br>** sensing：** `秒表`（Scratch 内置）或自定义计数变量 + `等待 1 秒` 循环<br>**外观：** `说 [内容] 2 秒`、`显示/隐藏`<br>**事件：** `当绿旗被点击`<br>**数据：** `将 [计时器] 设为 0`、`将 [计时器] 增加 1` |
| **预估课时** | **2 课时（90 分钟）** |
| **教学要点** | 1. 变量概念：用变量存储"时间"这个不断变化的值<br>2. 计时器实现原理：每秒增加一次，直到终点触发停止<br>3. `等待直到` 的用法：程序可以"等条件满足再继续"<br>4. 完整游戏循环：开始 → 过程 → 结束 → 反馈 → 重开<br>5. MVP 验证：此时整个赛车游戏已可玩 |
| **常见问题** | - 计时器不停走或走得不对 → 检查变量作用域（全局 vs 私有）<br>- `等待直到` 和 `重复执行` 搞混 → 用"排队等"vs"一直做"类比<br>- 到达终点后游戏没有停下来 → 缺少停止逻辑<br>- 分数/时间显示不出来 → 检查变量滑块是否拖到舞台上并设为可见<br>- 两课时之间衔接：第一课时完成计时逻辑，第二课时完成反馈和重赛 |

## 三、MVP 进度总表

| 阶段 | 内容 | 课时 | 累计 |
|------|------|------|------|
| 模块 1 | 赛车能移动 | 1 | 1 |
| 模块 2 | 有赛道 | 1 | 2 |
| 模块 3 | 起点和终点 | 1 | 3 |
| 模块 4 | 完整比赛（计时 + 反馈） | 2 | **5** |

> **总计：约 5 课时（4 × 45min + 1 × 90min）**  
> 第 5 课结束时，学生应能独立运行一辆赛车从起点开到终点并看到用时。
