## 背景与目标

### 现状

微信目前是澄木老师最主要的信息来源，承载了大量的工作沟通、项目信息、社交联络和个人事务。大脑同时处理微信、Telegram、飞书等多个渠道的信息，认知负荷很高。

### 短期约束

- 微信生态复杂，短期内不适合投入开发生产功能
- 所有微信相关任务默认 **Local Only**
- 目标是在不触碰红线的前提下，探索信息减负的可能性

### PoC 定义

本文档是可行性调研方案，不生成代码、不做生产部署。目的是回答"能做什么、风险在哪、优先级如何"，为后续决策提供依据。

---

## 红线与安全原则

以下事项在任何情况下都不做：

| # | 禁止事项 | 原因 |
|---|---------|------|
| 1 | 不做 Hook / 注入 | 违反微信 ToS，可能导致封号 |
| 2 | 不做抓包 | 技术风险和法律风险并存 |
| 3 | 不做协议模拟 | WeChat protocol reverse engineering 属于灰色地带 |
| 4 | 不做自动发消息 | 改变微信行为边界，触发风控 |
| 5 | 不上传微信原文到云端 | 隐私保护第一原则 |

**核心原则：** 微信数据不出本机。所有处理在本地完成，摘要后的结构化信息才可跨系统流转。

## 方向一：微信重要信息手动转发给小悟的半自动入口

### 概述

利用已有的 `@tencent-weixin/openclaw-weixin` 插件建立一条"反向通道"——澄木老师在微信里看到重要信息时，手动转发给已绑定的 OpenClaw 微信号，由小悟在本地进行处理。

### 操作流程设计

```
[澄木老师在看微信]
        │
        ▼
发现重要消息 → 长按该消息 → 转发给"小悟Bot"微信号
        │
        ▼
[OpenClaw WeChat Plugin 收到消息] ──(本地)──→ 小悟处理引擎
        │                                    │
        ▼                                    ▼
自动分类/打标                   生成摘要 / 提取待办 / 归档到 memory
        │                                    │
        ▼                                    ▼
[可选] 回复确认 ←───────────── 结构化结果存本地
```

### 具体实现思路（方案层面）

1. **入口建立：** OpenClaw 微信插件已支持扫码登录。澄木老师将小悟绑定为微信联系人，作为"信息收件箱"。
2. **手动触发：** 澄木老师在微信中看到需要记录/处理的信息时，长按消息 → 转发给小悟。这一步是半自动的核心——把判断权交给用户，机器只负责后续处理。
3. **本地处理流水线（在 OpenClaw 内）：**
   - 接收转发的文字/图片/文件
   - AI 自动分类：工作 / 个人 / 待办 / 参考材料
   - 提取关键信息生成摘要
   - 写入本地 memory 或 TODO 列表
4. **反馈回路：** 小悟可回复简短确认，如"✅ 已记录 — [项目名] 下周需要提交报告"。

### 能做到什么程度？

| 能力 | 状态 | 说明 |
|------|------|------|
| 文字消息转发处理 | ✅ 可行 | 插件原生支持，直接可用 |
| 图片 OCR + 理解 | ⚠️ 需评估 | 依赖本地 OCR 模型质量 |
| 文件接收和处理 | ⚠️ 需评估 | 文件大小和类型有限制 |
| 语音消息转文字 | ❌ 不推荐 | 涉及微信语音格式转换，复杂度高 |
| 群消息自动转发 | ❌ 不做 | 需要 Hook 或自动化操作，触碰红线 |

### 限制与风险点

- **依赖手动操作：** 核心弱点是仍需澄木老师主动转发。如果忘记转发，信息就漏了。这其实是设计上的取舍——宁可少记，不可错记/泄露。
- **微信插件稳定性：** 需要确认当前 `@tencent-weixin/openclaw-weixin` 插件在生产环境的连接稳定性和消息延迟情况。**👉 需要澄木老师确认当前微信插件的使用状态和体验。**
- **消息丢失风险：** 如果 OpenClaw Gateway 重启或网络抖动，转发中的消息可能丢失。
- **上下文丢失：** 转发的单条消息往往缺乏对话背景，AI 理解可能偏差。

### 建议优先级

**🟢 P0 — 最高优先级**  
理由：这是唯一不触碰任何红线的方案。利用已有插件基础设施，风险最低，可立即验证可行性。即使最终其他方向不可行，这仍然是一个有效的降级方案。

### 下一步动作

1. [ ] **澄木老师确认：** 当前 OpenClaw 微信插件是否已启用？日常使用体验如何？
2. [ ] 设计"转发消息格式约定"——例如澄木老师可以在转发时加前缀来标记优先级，如 `!重要`、`#待办`、`@参考`
3. [ ] 在 OpenClaw 中配置消息处理规则（不写代码，先用 config 层面测试）
4. [ ] 小范围灰度测试：澄木老师试用 1 周，评估漏记率和误分类率

## 方向二：macOS / Windows 微信本地数据只读 PoC

### 概述

微信客户端在本地磁盘以 SQLite 数据库形式存储聊天记录、联系人、文件等数据。研究方向是能否安全地进行只读访问，提取结构化信息供小悟本地处理。

### 微信本地数据存储现状（基于调研）

**存储架构：**

| 项目 | macOS (微信 4.x) | Windows (微信 4.x) |
|------|-------------------|---------------------|
| 数据库引擎 | SQLCipher 4 / WCDB | SQLCipher 4 / WCDB |
| 加密算法 | AES-256-CBC | AES-256-CBC |
| 密钥派生 | PBKDF2-HMAC-SHA512 (256,000 轮) | 同左 |
| 页面大小 | 4096 bytes | 4096 bytes |
| 每个 DB 有独立 Salt | ✅ | ✅ |

**主要数据库文件：**
- `MSG.db` / `MicroMsg.db` — 聊天记录和联系人
- `MediaMSG.db` — 媒体消息元数据
- `OpenIMContact.db` — 公众号/企业微信联系人
- `Favorite.db` — 收藏内容
- `Sns.db` — 朋友圈相关

**macOS 路径（典型）：** `~/Library/Containers/netease.music/Data/...`  
实际上微信 macOS 版使用 App Sandbox，数据位于应用的沙盒容器内。  
**Windows 路径（典型）：** `%USERPROFILE%\Documents\WeChat Files\[wxid]\DB\`

### 加密挑战分析

这是该方向的最大难点：

1. **SQLCipher 4 加密：** 所有数据库文件都经过 AES-256-CBC 加密，直接用 SQLite 浏览器打不开。
2. **每库独立密钥：** 每个 `.db` 文件有独立的 32-byte salt，需要分别解密。
3. **密钥在内存中：** WeChat 使用 WCDB（腾讯自研的 SQLCipher 封装）时，派生后的原始密钥会被缓存在进程内存中，格式为 `x'<64hex_enc_key><32hex_salt>'`（80 字符 hex 串）。
4. **内存提取工具已存在：** GitHub 上已有多个项目实现了从微信进程内存中提取加密密钥并解密数据库的功能（如 `wechat-decrypt`、`wx-cli` 等），支持 Windows 和 macOS。

**⚠️ 但存在一个关键问题：这些工具通过读取/扫描 WeChat 进程内存来提取密钥。虽然目的是"只读"，但技术实现涉及对微信进程的介入，可能触发以下风险。**

### 法律与安全风险评估

| 风险维度 | 级别 | 说明 |
|----------|------|------|
| **违反 ToS** | 🔴 高 | 微信用户协议明确禁止逆向工程和数据提取。即使只读访问自己的数据，也处于灰色地带。 |
| **账号封禁** | 🟡 中 | 微信有反作弊/风控系统。内存扫描或注入可能被检测到，导致账号受限。 |
| **稳定性依赖** | 🔴 高 | 微信版本更新频繁。每次更新都可能改变加密方案、数据库 schema、内存布局，导致工具失效。需要持续维护。 |
| **数据安全** | 🟡 中 | 解密后的明文数据库包含大量个人通信记录。存储和访问控制不当可能导致隐私泄露。 |
| **macOS SIP / 权限** | 🟠 中高 | macOS 系统完整性保护 (SIP) 可能阻止对微信进程内存的读取，需要关闭某些保护或授予特殊权限，降低系统安全性。 |

### 可行性结论

| 维度 | 评分 (1-5) | 说明 |
|------|-----------|------|
| 技术可行性 | 3/5 | 理论可行，工具链存在，但需持续适配微信更新 |
| 法律合规性 | 1/5 | ToS 风险显著 |
| 账号安全风险 | 2/5 | 有封号可能 |
| 维护成本 | 1/5 | 每次微信更新都可能导致失效 |
| 信息收益 | 4/5 | 如果成功，可以获得完整的历史消息数据 |

**综合评分：⭐ 2.2 / 5 — 技术可行性一般，风险偏高，不建议作为 PoC 首选。**

### 建议优先级

**🔴 P3 — 低优先级（不推荐在 PoC 阶段尝试）**  
理由：虽然技术上部分可行，但法律风险、封号风险和维护成本使得这一方向不适合作为第一步。除非其他所有方向都被排除，否则不建议投入资源。

### 如坚持探索的最小化方案

如果澄木老师认为本地数据访问是必须的，最小化 PoC 方案：

1. **严格只读：** 绝不修改微信的任何文件、内存或行为
2. **完全离线：** 解密过程在本机完成，不在任何网络上传输原始数据
3. **增量快照：** 定期（如每周）做一次快照导出，不实时扫描
4. **限定范围：** 只查询特定关键词/时间段的消息，不全量导出
5. **手动触发：** 每次解密都由澄木老师手动启动，不自动化

### 下一步动作

1. [ ] **澄木老师确认：** 是否对本地微信数据访问有强需求？主要想解决什么场景？
2. [ ] 如果确认需要，先在一台非主力机器上做技术验证（不使用主力微信号）
3. [ ] 研究最新微信版本的加密变化（微信 4.x 仍在快速迭代加密方案）

## 方向三：Telegram 收件箱自动整理

### 概述

与微信不同，Telegram 提供了完善的 Bot API。OpenClaw 已接入 Telegram 渠道，可以直接处理收到的消息。这个方向的可行性最高。

### 当前状态

- OpenClaw 已配置 Telegram 渠道（`telegram-pairing.json` 存在）
- Telegram 消息直接到达小悟会话中
- 目前缺少自动分类和摘要机制

### 可以做哪些自动化整理？

#### 3.1 消息路由与优先级标记

**方案：** 在 OpenClaw 的消息处理层添加一个"分类网关"，在消息进入主会话前自动打标：

```
[Telegram 消息到达]
        │
        ▼
   分类引擎 ──┬── 🔴 紧急（直接推送给澄木老师）
             ├── 🟡 重要（加入待处理队列）
             ├── ⚪ 普通（纳入日常摘要）
             └── ⚫ 噪音（自动归档，不通知）
```

**分类依据：**
- 发送者身份（联系人 vs 群聊 vs 频道）
- 消息内容关键词（"urgent"、"deadline"、"明天"等）
- 消息类型（文字 / 文件 / 链接 / 图片）
- 历史互动模式（经常互动的联系人自动高优先级）

#### 3.2 群组消息摘要

**方案：** 对于高频群聊，不逐条推送，而是每隔固定时间（如每 2 小时）生成一份摘要：

```
📋 [项目讨论群] 摘要 (14:00-16:00)
━━━━━━━━━━━━━━━━━━━━━━━━━━
• @张三: 提出了 API 设计方案变更
• @李四: 确认了前端排期，下周三前完成
• @王五: 分享了性能测试报告链接
• 决策点: API v2 的认证方式待定 ⚠️
━━━━━━━━━━━━━━━━━━━━━━━━━━
涉及你吗？ → [快速标记]
```

#### 3.3 频道/广播消息过滤

**方案：** Telegram 频道消息量大但信息密度低，可以：
- 自动忽略纯转发/无评论的消息
- 只保留有明确行动项或高价值内容的帖子
- 按主题分类推送

#### 3.4 "消化时间"模式

**方案：** 设置一个每日固定时段（如晚上 8 点），在这个时间点统一处理积压的 Telegram 消息，而不是逐条打扰：

```
🌙 晚间 Telegram 汇总 (20:00)
━━━━━━━━━━━━━━━━━━━━━━━━━━
新消息: 47 条 | 需回复: 3 条 | 待办提取: 2 项
━━━━━━━━━━━━━━━━━━━━━━━━━━

需要立即关注:
1. [项目A] @张三 询问部署进度
2. [个人] 妈妈发了生日邀请  

可延后处理:
- 技术交流群讨论 (15条，无行动项)
- 新闻频道推送 (12条)
```

### 限制与风险点

| 风险 | 级别 | 说明 |
|------|------|------|
| Bot API 速率限制 | 🟢 低 | Telegram Bot API 限流宽裕，日常使用不会触发 |
| 消息处理延迟 | 🟢 低 | OpenClaw 通道成熟，延迟可接受 |
| 分类准确性 | 🟡 中 | AI 分类可能存在误判，需要反馈机制持续优化 |
| 隐私边界 | 🟢 低 | Telegram 消息已在 OpenClaw 会话内处理，不涉及额外的数据传输 |

### 建议优先级

**🟢 P0 — 最高优先级**  
理由：Telegram API 完善、风险极低、OpenClaw 已接入。这是性价比最高的减负方案，可以立即开始设计分类规则和摘要模板。

### 下一步动作

1. [ ] **澄木老师确认：** 目前 Telegram 消息量级？大概每天多少条？主要分布在哪些群/频道？
2. [ ] 分析最近一周的 Telegram 消息模式（发送者分布、类型分布、互动频率）
3. [ ] 设计分类规则初稿和摘要模板
4. [ ] 在 OpenClaw config 中配置消息处理 pipeline（非代码层面，用现有配置机制）

## 方向四：每日早晚摘要方案

### 概述

从各渠道（微信转发、Telegram、飞书）收集信息后，生成结构化的早晚两份摘要。目标是让澄木老师用最少的时间了解所有需要关注的事项。

### 摘要输入源

| 来源 | 数据获取方式 | 实时性 |
|------|-------------|--------|
| 微信 | 手动转发到小悟（方向一） | T+0 ~ T+N（取决于转发频率） |
| Telegram | Bot API 自动收集 | 接近实时 |
| 飞书 | OpenClaw 飞书插件 / API | 近实时 |
| 其他 | 邮件、日历等 | 定时拉取 |

### 早间摘要设计（建议时间：08:00）

### 摘要格式设计原则

为了让摘要"最省力"（一眼扫完、无需二次加工）：

1. **层次分明：** 紧急 → 重要 → 一般 → 噪音，严格按优先级排序
2. **行动导向：** 每条信息都标注是否需要行动、截止时间和优先级
3. **可折叠细节：** 先看摘要，需要详情再展开
4. **数字锚定：** 用具体数量（"3条需回复"而非"有几条消息"）降低认知模糊
5. **模板一致性：** 每天格式相同，让大脑形成条件反射式阅读习惯

### 限制与风险点

| 风险 | 级别 | 说明 |
|------|------|------|
| 信息遗漏 | 🟡 中 | 微信手动转发有漏记可能，摘要质量取决于输入完整性 |
| AI 判断偏差 | 🟡 中 | 优先级排序和摘要提取依赖 AI 准确度 |
| 过度简化 | 🟠 中高 | 复杂问题可能被摘要简化掉关键背景信息 |
| 生成成本 | 🟢 低 | Ollama 本地模型生成摘要的成本可控 |

### 建议优先级

**🟡 P1 — 高优先级（在方向一、三的基础上实施）**  
理由：摘要方案依赖于前面方向的输入管道搭建完成。一旦 Telegram 整理和微信手动转发入口就绪，摘要可以很快跟进。

## 方向五：从消息中提取待办事项

### 概述

在日常信息流中自动识别和提取待办事项，是降低认知负荷的核心功能。目标是在不增加澄木老师操作负担的前提下，将隐含的"需要做的事"显式化为结构化待办列表。

### 什么模式的消息应该被提取为待办？

#### 自动识别模式

| 模式类型 | 示例 | 置信度 |
|----------|------|--------|
| **明确指令** | "明天把报告发我"、"周五之前交方案" | 🔴 高 |
| **截止日期暗示** | "下周一开会，记得准备 PPT" | 🟠 中高 |
| **疑问需跟进** | "那个 API 的问题你看了吗？" | 🟡 中 |
| **会议/活动通知** | "下午3点讨论新功能，别忘了" | 🔴 高 |
| **承诺型语句** | "好的我来处理"、"没问题我负责" | 🟠 中高 |
| **任务委派** | "你帮我看看这份文档" | 🔴 高 |

#### 不应自动提取的模式

| 模式类型 | 示例 | 原因 |
|----------|------|------|
| 纯信息分享 | "看到这个文章，挺有意思的" | 非行动项 |
| 寒暄/社交 | "最近怎么样？"、"周末愉快！" | 非工作相关 |
| 已解决的讨论 | "哦那没问题了，继续吧" | 无待执行内容 |

### 低门槛标记和提取方式

#### 方案 A：AI 自动提取（推荐）

**流程：** 消息经过 OpenClaw 处理引擎时，自动扫描是否有待办语义。如果检测到，自动创建待办条目并请求确认：

```
🤖 检测到潜在待办事项：

"明天把报告发我" (来自: 张三, 微信)
→ 提取为: [待办] 发送报告给张三 | 截止: 明天 | 来源: 微信

✅ 确认提取   ❌ 忽略此条
```

**优点：** 零操作成本，澄木老师只需偶尔纠正误判。  
**缺点：** AI 可能过度提取（把"好的"也当成待办），需要一段时间的训练期。

#### 方案 B：快捷标记语法

在方向一的微信手动转发中约定简单的标记语法：

| 标记 | 含义 | 示例 |
|------|------|------|
| `!` | 重要/需立即关注 | 转发消息时附加 `!这个很急` |
| `#待办` | 明确标记为待办 | `#待办 下周三前完成 API 文档` |
| `?` | 需要澄木老师自己判断 | `?这份文件要不要处理？` |
| `@参考` | 存档参考，非行动项 | `@参考 行业趋势报告2026Q2` |

**优点：** 精确可控，澄木老师完全掌握。  
**缺点：** 增加了手动操作的认知负担。

#### 方案 C：混合模式（推荐最终方案）

AI 自动提取作为第一层过滤器 + 快捷标记语法用于特殊情况 + 定期人工审核确认：

```
┌─────────────────────────────────────┐
│         待办提取流水线              │
│                                     │
│ 消息到达 → AI 扫描 → 检测到行动语义？ │
│           │           ├─ YES → 创建草稿待办 │
│           │           └─ NO  → 正常处理     │
│                                           │
│         草稿待办池                      │
│   [每日早晚摘要中展示，等待确认]        │
│           │                              │
│           ├─ 澄木老师✅ → 正式待办       │
│           └─ 澄木老师❌ → 忽略 + 学习    │
└─────────────────────────────────────┘
```

### 待办存储与同步

| 项目 | 方案 | 说明 |
|------|------|------|
| 存储位置 | 本地文件 (`memory/todays-todo.md`) | Local Only，不上传云端 |
| 格式 | Markdown checklist | 简单、可移植、人类可读 |
| 状态追踪 | ✅ / ⏳ / ❌ | 完成 / 进行中 / 取消 |
| 与摘要联动 | 自动出现在早晚摘要中 | 无缝集成到方向四的摘要流 |

### 建议优先级

**🟡 P1 — 高优先级（与方向四并行实施）**  
理由：待办提取是信息减负的核心收益点。可以在 Telegram 消息处理管道（方向三）中先行实现，再扩展到微信手动转发管道（方向一）。

### 下一步动作

2. [ ] 设计 AI 待办提取的 prompt 模板
